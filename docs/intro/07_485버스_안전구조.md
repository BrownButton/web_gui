# 07. RS-485 버스 안전 구조

## 1. 문제: 버스 충돌

RS-485는 **반이중(Half-Duplex)** 통신입니다.
한 순간에 하나의 장치만 버스를 점유하여 데이터를 전송할 수 있습니다.

### 충돌이 발생하는 상황

이 앱은 주기적으로 모든 장치를 순차 폴링합니다.
폴링 중에 사용자가 버튼을 클릭하여 Setpoint를 변경하면:

```
시간 →

폴링 루프:  [TX: Slave1 요청] ──→ [응답 대기 중 ...]
                                        ↑
사용자 클릭: ────────────────── [TX: Setpoint 쓰기] ← ⚠️ 충돌!
```

두 TX가 동시에 버스에 올라오면:
- 양쪽 프레임이 모두 손상됨
- 슬레이브가 알 수 없는 프레임 수신 → 응답 없음
- "Response timeout" 오류 발생
- 심각한 경우 통신 불능 상태

---

## 2. 해결: Command Queue 패턴

사용자 명령을 **큐(Queue)**에 쌓아놓고,
폴링 사이클 **사이**에 안전하게 처리합니다.

```
시간 →

폴링 루프:  [TX: Slave1 요청] → [응답 수신] ─ [큐 처리] ─ [TX: Slave2 요청] → ...
                                                    ↑
사용자 클릭: ─────────────────→ commandQueue에 추가 → 여기서 안전하게 처리됨
```

---

## 3. 상태 변수

| 변수 | 타입 | 역할 |
|------|------|------|
| `autoPollingTimer` | 타이머 참조 | 폴링이 활성화되어 있는지 (null이면 비활성) |
| `isPolling` | boolean | 현재 폴링 사이클 진행 중 여부 |
| `isScanning` | boolean | 장치 스캔 진행 중 여부 |
| `commandQueue` | Array | 대기 중인 사용자 명령 목록 |

### 상태 조합의 의미

| autoPollingTimer | isPolling | 의미 |
|-----------------|-----------|------|
| null | false | 폴링 비활성 (연결 안 됨 또는 수동 모드) |
| 있음 | false | 폴링 활성, 현재 사이클 완료 후 대기 중 → **사용자 명령을 큐에 추가** |
| 있음 | true | 폴링 사이클 실행 중 → **절대 직접 TX 금지** |

---

## 4. 안전한 API 경로

### 레지스터 쓰기: `writeRegister()`

```javascript
async writeRegister(slaveId, address, value) {
    const frame = ModbusRTU.buildWriteSingleFrame(slaveId, address, value);

    if (this.autoPollingTimer && !this.isPolling) {
        // 폴링 활성이지만 사이클 사이 → 큐에 추가
        return new Promise((resolve, reject) => {
            this.commandQueue.push({ frame, slaveId, address, resolve, reject });
        });
    } else {
        // 폴링 비활성 → 직접 전송
        return this.sendWriteAndWaitResponse(frame, slaveId);
    }
}
```

### 레지스터 읽기: `readRegisterWithTimeout()`

```javascript
async readRegisterWithTimeout(slaveId, address) {
    const frame = ModbusRTU.buildReadHoldingFrame(slaveId, address, 1);

    if (this.autoPollingTimer && !this.isPolling) {
        // 폴링 활성이지만 사이클 사이 → 큐에 추가
        return new Promise((resolve, reject) => {
            this.commandQueue.push({
                type: 'read', frame, slaveId, address, resolve, reject
            });
        });
    } else {
        // 직접 전송 (폴링 루프 내부 또는 폴링 비활성)
        return this.sendAndWaitResponse(frame, slaveId);
    }
}
```

## 5. 폴링 사이클 다이어그램

```
pollNextDeviceSequential() 실행 흐름:

startAutoPolling()
  │
  └→ setInterval(poll, 0)  ← 사이클 반복
       │
       ↓
     isScanning? ── Yes → skip (스캔 중이면 폴링 건너뜀)
       │ No
       ↓
     isPolling = true   ← ⚠️ 이 구간에서 절대 직접 TX 금지
       │
       ↓
     For each device:
       │
       ├→ sendAndWaitResponse(FC04, 0xD011)  ← Motor Status
       │     ↓ (응답 or 타임아웃)
       │  device.status 업데이트
       │
       ├→ sendAndWaitResponse(FC04, 0xD02D)  ← Actual Speed
       │     ↓
       │  device.actualSpeed 업데이트
       │
       └→ (모니터링 파라미터 폴링)
       │
       ↓
     isPolling = false   ← ⚠️ 이제 안전
       │
       ↓
     [commandQueue 처리]  ← 사용자 명령 여기서 실행
       │
       ├→ queue[0] 꺼내서 처리
       ├→ queue[1] 꺼내서 처리
       └→ queue 비워질 때까지 반복
       │
       ↓
     다음 사이클로
```

## 6. 장치 스캔 시 특별 처리

`isScanning = true`로 설정해도 **이미 진행 중인 폴링 사이클은 막을 수 없습니다**.
따라서 스캔 시작 전에 반드시 현재 사이클 완료를 기다려야 합니다:

```javascript
async scanWithTimeout(slaveId) {
    this.isScanning = true;     // 새 폴링 사이클 시작 방지

    // ⚠️ 중요: 이미 진행 중인 사이클이 있을 수 있음
    while (this.isPolling) {
        await delay(5);          // 5ms씩 대기
    }
    // 이제 isPolling = false → 안전하게 TX 가능

    const frame = ModbusRTU.buildReadInputFrame(slaveId, 0xD011, 1);
    return await this.sendAndWaitResponse(frame, slaveId);
}
```

## 7. 절대 금지 항목

다음 함수들은 **폴링 루프 내부에서만** 호출해야 합니다.
새 기능 추가 시 절대 직접 호출하지 마세요:

```javascript
// ❌ 금지: 폴링 중 직접 전송
this.writer.write(frame)               // RS-485 직접 쓰기
this.sendAndWaitResponse(frame, id)    // 내부 전용 함수
this.sendWriteAndWaitResponse(frame, id) // 내부 전용 함수
```

## 8. 새 기능 추가 시 체크리스트

새로운 기능에서 RS-485 통신이 필요할 때:

### ✅ 체크리스트

- [ ] **레지스터 읽기** → `readRegisterWithTimeout()` 또는 `readInputRegisterWithTimeout()` 사용
- [ ] **레지스터 쓰기** → `writeRegister()` 사용
- [ ] **직접 `writer.write()` 호출** → 절대 금지
- [ ] **스캔 중 TX 필요** → `while (isPolling) await delay(5)` 선행

### ✅ 올바른 코드 예시

```javascript
// ✓ 올바름: 안전한 API 사용
async myNewFeature(device) {
    // 읽기
    const speed = await this.readInputRegisterWithTimeout(
        device.slaveId, 0xD02D
    );

    // 쓰기
    await this.writeRegister(device.slaveId, 0xD001, newSetpoint);
}
```

### ❌ 잘못된 코드 예시

```javascript
// ✗ 잘못됨: 직접 TX 호출 (버스 충돌 발생!)
async myBadFeature(device) {
    const frame = ModbusRTU.buildReadInputFrame(device.slaveId, 0xD02D, 1);
    await this.writer.write(frame);  // ← 위험!
}
```

## 9. 통신 성공률 통계

버스 충돌 방지 구조 덕분에 통신 성공률을 모니터링할 수 있습니다.

```javascript
// 각 장치의 통신 통계
device.stats = {
    requests: 1000,   // 총 요청 수
    success:  998,    // 성공 (응답 수신)
    errors:   2,      // 오류 (CRC 불일치 등)
    failures: 0       // 연속 타임아웃 (오프라인 카운터)
};

// 상단 NavBar에 표시되는 전역 통계
this.stats = {
    requests: number,     // 전체 요청 수
    success:  number,     // 전체 성공 수
    errors:   number      // 전체 오류 수 (에러율 = errors/1000 프레임)
};
```

대시보드 상단의 통계 표시:
```
[요청: 12,450]  [성공: 12,448]  [에러율: 0/1000]
```

에러율이 높아질 경우 버스 충돌이나 케이블 문제를 의심해볼 수 있습니다.
