# 합격조건 문제점 분석

**작성일**: 2026-03-04
**대상**: OS 검증 리스트 전체 항목 (INDEX.md 기준)

---

## 개요

이 문서는 각 검증 항목의 판정기준(합격조건) 중 논리적으로 모순되거나, 실제 검증이 불가능하거나, 판정 주체에 따라 결과가 달라질 수 있는 항목들을 정리한다.

**분류 기준**

| 분류 | 설명 |
|------|------|
| 🔴 자기모순 | 같은 문서 내에서 조건이 서로 충돌 |
| 🟠 미완성 | 시험 방법 자체가 미확정/미구현 |
| 🟡 주관적 기준 | 수치 없이 주관적 판단에 의존 |
| 🟣 검증 불일치 | 판정기준이 실제 목적과 다른 것을 측정 |
| 🔵 필수 정보 누락 | 주소·파라미터 등 시험에 필수적인 값이 없음 |
| ⚫ 재현 불가 | 시험 조건 자체가 불안정하거나 비재현적 |

## 1. 🔴 자기모순 (Self-contradictory)

### 2-2. FC(04) 동작 시험 — 판정 기준이 스스로를 부정

> **파일**: [2-2. FC(04) 동작 시험.md](2.%20Modbus%20RTU/2-2.%20FC(04)%20동작%20시험.md)

**문제**

판정기준에 `Value = 0x4242` 라고 특정 값을 명시해 놓고, 바로 같은 문서의 시험 방법에서:

> "현재 드라이브의 상태에 따라 위의 데이터와 **상이할 수 있음**"

이라고 스스로 부정하고 있다.

- 0xD000 (FC04로 읽는 Identification 레지스터)이 항상 0x4242를 반환한다는 근거가 없다.
- 드라이브 상태에 따라 값이 달라진다면 0x4242는 판정 기준이 될 수 없다.

**올바른 방향**
FC04 프로토콜 동작 확인이 목적이라면, 판정기준은 "응답 프레임의 형식이 Modbus RTU 규격에 맞는지" 또는 "CRC 오류 없이 응답이 수신되는지"가 되어야 한다.

## 2. 🟠 미완성 — 시험 방법이 미확정 또는 미구현

### 5-1. Error 0x10 IPM Fault — 유발 방법 미정

> **파일**: [5-1. Error 0x10 IPM fault.md](5.%20보호동작/5-1.%20Error%200x10%20IPM%20fault.md)

**문제**

시험 방법 step 4에:

> "(IPM Fault 유발 방법은 **개발팀 협의 후 결정**)"

이라고 명시되어 있다. 시험 방법이 확정되지 않았으므로 판정기준이 있어도 시험 자체를 진행할 수 없다. 검증서로서 성립하지 않는다.

### 5-3. Error 0x14 Over Current — 구현 예정 단계

> **파일**: [5-3. Error 0x14 over current.md](5.%20보호동작/5-3.%20Error%200x14%20over%20current.md)

**문제**

시험 방법에:

> "📌 프로시저(Procedure)에 할당해서 처리하는 방식으로 **구현 예정**"

이라고 명시되어 있다. 유발 프로시저가 아직 구현되지 않았으므로 시험 불가. 이 항목은 현재 검증 대상이 아니다.

## 3. 🟡 주관적 기준 — 수치 없는 합격 판정

### 3-6. Board 온도 / 3-7. IGBT 온도 모니터링 — "합리적인 값"

> **파일**: [3-6. Board 온도.md](3.%20기본동작/3-6.%20Board%20온도.md), [3-7. IGBT 온도.md](3.%20기본동작/3-7.%20IGBT%20온도.md)

**문제**

판정기준:
> "Board 온도 파라미터가 **합리적인 온도값**을 출력함"

"합리적인"이 무엇인지 정의가 없다. 시험자마다 판정이 달라진다.

- 파라미터의 단위(°C? 0.1°C? raw count?)가 명시되지 않음
- 상온 조건에서 허용 범위(예: 15~40°C ± 5°C)가 없음
- 기준 온도 측정 장비와 파라미터 값의 비교 허용 오차가 없음

**올바른 방향**
예: "상온(25°C ± 5°C) 환경에서 Board 온도 파라미터가 실측값과 ±5°C 이내로 일치할 경우 합격"

---

### 3-10-1. Main OS 다운로드 — LED 점멸 주기 육안 확인

> **파일**: [3-10-1. main OS 다운로드.md](3.%20기본동작/3-10-1.%20main%20OS%20다운로드.md)

**문제 1 — 버전 미특정**

판정기준:
> "펌웨어 버전이 **업데이트된 버전**으로 확인됨"

어떤 버전을 다운로드하는지, 업데이트 후 어떤 버전이 되어야 하는지 명시가 없다. 시험 결과 표의 "기대 버전" 칸도 공란(-)이다.

**문제 2 — LED 점멸 주기**

> "(Main OS의 경우 **LED 점멸 주기 변화**를 통해 업데이트 확인)"

점멸 주기가 얼마에서 얼마로 바뀌어야 하는지 기준이 없다. 육안으로 "변화"를 확인하는 것은 객관적 기준이 아니다.

**올바른 방향**
다운로드할 목표 버전 명시 + 버전 레지스터(Main SW Version, Inverter SW Version)로 수치 확인을 주 판정기준으로 삼아야 한다.

## 4. 🟣 검증 불일치 — 목적과 다른 것을 측정

### 4-6. Closed-loop Velocity Control — 지령값만 확인하고 추종 여부는 부차적

> **파일**: [4-6. Closed-loop Velocity Control.md](4.%20구동동작/4-6.%20Closed-loop%20Velocity%20Control.md)

**문제 1 — Command Velocity vs Actual Speed 우선순위**

판정기준:
> "0xD051 (Command Velocity) 파라미터의 값 **100** 확인 (Velocity Command 100rpm 출력 확인)"

Command Velocity [0xD051]는 드라이브가 내부적으로 처리한 **지령값**이다. 이 값이 100이라는 것은 RS485로 보낸 Set Point가 제대로 수신·변환되었음을 뜻할 뿐, **모터가 실제 100rpm으로 회전하고 있다는 증거가 아니다**.

Closed-loop 제어의 핵심 검증은 실제 속도(Actual Speed [0xD02D])가 목표 속도를 추종하는지 여부인데, 이 항목이 부가 확인(step 7)으로 처리되고 판정기준에는 없다.

**문제 2 — Set Point 3200 = 100rpm 계산 불일치**

> "Set Point를 **3200**으로 설정한다 (100rpm)"

속도 제어 변환식: `raw = rpm / maxSpeed × 64000`
3200 = rpm / maxSpeed × 64000 → **maxSpeed = 2000rpm**이 되어야 100rpm이다.

현재 시스템에 maxSpeed가 1600rpm으로 사용 중이라면, Set Point 3200은 **80rpm**에 해당한다. 100rpm이 아니다. 판정기준의 "D051 값 = 100"도 사실은 80이 나올 가능성이 높다.

**올바른 방향**
- maxSpeed 확인 후 Set Point 재계산 필요
- 판정기준 주 항목을 Actual Speed [0xD02D]의 목표 추종 (예: ±5rpm 이내)으로 변경

### 5-19. Error 0x53 Excessive Speed Deviation — 5-20과 조건 중복

> **파일**: [5-19. Error 0x53 excessive speed deviation.md](5.%20보호동작/5-19.%20Error%200x53%20excessive%20speed%20deviation.md)

**문제**

시험 방법:
> "팬을 **물리적으로 구속**하거나 부하를 가하여 실제 속도가 목표 속도를 따라가지 못하는 조건을 만든다"

팬을 물리적으로 완전히 구속하면 5-20 (Motor Shaft Blocked, Error 0x58)이 먼저 발생할 가능성이 높다. 그 경우 0x53이 발생하기 전에 드라이브가 이미 정지한다.

판정기준에 명시된 0x53 에러가 실제로 발생한다는 보장이 없으며, 5-20과 동일한 시험 방법을 쓰면서 다른 에러가 목표다.

**올바른 방향**
0x53은 완전 구속이 아닌 **부분 부하(외부 팬 등 추가 관성)**로 속도 편차만 발생시키는 조건이어야 한다. 또는 속도 편차 임계값 파라미터를 낮게 설정하는 방법 등을 사용해야 한다.

## 5. 🔵 필수 정보 누락 — 주소 또는 파라미터 미기재

### 3-5. DCLink V 모니터링 — 레지스터 주소 공란

> **파일**: [3-5. DCLink V.md](3.%20기본동작/3-5.%20DCLink%20V.md)

**문제**

요청 프레임 표의 주소가:
```
DC Link Addr Hi | DC Link Addr Lo
```
로 실제 주소가 빠져 있다. 이 검증서만 보고는 시험을 진행할 수 없다.

판정기준 "DC Link 전압 파라미터가 실제 DC 전압값을 정상적으로 반영함"도 허용 오차 없이 주관적이다. DC Link 전압이 어느 범위 안에 있어야 합격인지 기준이 없다.

### 5-27. Warning 0x40 Torque Limit — Warning Status 주소 공란

> **파일**: [5-27. Warning 0x40 Torque Limit.md](5.%20보호동작/5-27.%20Warning%200x40%20Torque%20Limit.md)

**문제**

Warning Status 파라미터 읽기 프레임:
```
Warning Addr Hi | Warning Addr Lo
```
실제 Modbus 레지스터 주소가 기재되지 않았다. 어느 레지스터를 읽어야 하는지 알 수 없어 시험 진행이 불가능하다.

## 6. ⚫ 재현 불가 — 비재현적 시험 환경

### 5-20. Error 0x58 Motor Shaft Blocked — 테이프 구속

> **파일**: [5-20. Error 0x58 motor shaft is Blocked.md](5.%20보호동작/5-20.%20Error%200x58%20motor%20shaft%20is%20Blocked.md)

**문제**

시험 방법:
> "팬을 **테이프 등으로** 회전하지 못하도록 구속시킨다"

시험 Set Point는 10% 토크(6554)인데, 테이프의 강도에 따라:
- 테이프가 약하면 → 모터가 돌아버려 에러 미발생
- 테이프가 강하면 → 구속 성공, 에러 발생

테이프 종류·길이·감은 횟수를 특정하지 않으면 결과가 달라진다. 재현성이 없다. 또한 테이프가 모터에 달라붙어 이물질 오염을 일으킬 수 있다.

**올바른 방향**
전용 구속 지그(jig)를 제작하거나, 소프트웨어적으로 에러를 주입하는 프로시저 방식을 사용해야 한다.
