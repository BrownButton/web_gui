## 4. 데이터 흐름 다이어그램

### 폴링 데이터 흐름

```
┌──────────────────────────────────────────────────────┐
│                    app.js                            │
│                                                      │
│  startAutoPolling()                                  │
│    ↓                                                 │
│  pollNextDeviceSequential()    ← 100ms 간격 반복      │
│    ↓                                                 │
│  For each device:                                    │
│    ↓                                                 │
│  modbus.js:buildReadInputFrame(slaveId, 0xD011)      │
│    ↓                                                 │
│  writer.write(frame)  → [RS-485 버스] → EC-FAN       │
│                                            ↓         │
│  reader.read()        ← [RS-485 버스] ← 응답 데이터  │
│    ↓                                                 │
│  modbus.js:parseResponse()                           │
│    ↓                                                 │
│  device.actualSpeed = parsedValue                    │
│  device.status = 'online'                            │
│    ↓                                                 │
│  updateDeviceCard(device)  → DOM 업데이트 → 화면 갱신 │
└──────────────────────────────────────────────────────┘
```

### 사용자 쓰기 명령 흐름

```
사용자: 슬라이더 조정
  ↓
writeRegister(slaveId, 0xD001, newValue)
  ↓
  ├── (폴링 중이면) commandQueue에 추가 ──→ 폴링 사이클 사이에 처리
  └── (폴링 없으면) 즉시 전송 ──────────→ 응답 수신 → Promise 반환
```

## 5. Device 객체 구조

각 EC-FAN 장치는 다음 구조의 JavaScript 객체로 표현됩니다.
C# 클래스와 비교하면 이해하기 쉽습니다:

```javascript
// JavaScript 객체 (C# class와 유사)
{
  id: "uuid-1234",           // 내부 고유 식별자 (UUID)
  slaveId: 1,                // Modbus Slave ID (1~247)
  name: "EC-FAN #1",         // 표시 이름

  // 실시간 상태
  status: "online",          // "online" | "offline"
  actualSpeed: 1200,         // 현재 속도 (RPM)
  setpoint: 1600,            // 목표 설정값 (raw)

  // 설정값
  operatingMode: 0,          // 0=속도제어, 2=개루프
  direction: 0,              // 0=CCW, 1=CW
  maxCoilCurrent: 100,       // 최대 코일 전류

  // 모니터링 파라미터 (선택한 레지스터 목록)
  monitoringParams: [
    { address: 0xD02D, name: "Actual Speed", value: 1200, unit: "RPM" },
    { address: 0xD011, name: "Motor Status", value: 1, unit: "" }
  ],

  // 통계
  stats: {
    requests: 1000,          // 총 요청 수
    success: 998,            // 성공 수
    errors: 2,               // 오류 수
    failures: 0              // 연속 타임아웃 수
  },

  lastUpdate: 1709612400000  // 마지막 응답 시각 (ms)
}
```

## 6. 초기화 흐름

앱 시작 시 다음 순서로 초기화됩니다:

```
브라우저에서 index.html 열기
  ↓
HTML 파싱 완료 (DOMContentLoaded 이벤트)
  ↓
new ModbusDashboard()  ← 메인 클래스 인스턴스 생성
  ↓
  ├── initializeUI()          ← 버튼, 탭 이벤트 리스너 등록
  ├── loadParameters()        ← parameters.csv 파싱 → 파라미터 목록 구성
  ├── loadSettings()          ← localStorage에서 앱 설정 복원
  ├── loadPollingSettings()   ← 폴링 간격 복원
  ├── loadDevices()           ← localStorage에서 장치 목록 복원
  └── initializeChartPage()   ← 차트 캔버스 초기화

  ↓
window.dashboard = instance   ← 전역 접근 가능하게 등록
  ↓
  ← 사용자 입력 대기 (이벤트 드리븐)
```

## 7. parameters.csv 역할

모든 레지스터 파라미터 정의가 CSV 파일에 저장되어 있습니다.

```csv
Address,Name,Unit,Min,Max,Access,Description
0xD001,Setpoint,,0,65535,RW,Speed setpoint
0xD011,Motor Status,,0,1,R,0=Stop 1=Run
0xD02D,Actual Speed,RPM,0,3000,R,Current motor speed
0xD100,Fan Address,,1,247,RW,Modbus slave ID
```

이 파일을 기반으로:
- Parameters 탭에 레지스터 목록 자동 생성
- 모니터링 파라미터 선택 UI 자동 생성
- 최소/최대값 유효성 검사

새 파라미터 추가 시 CSV만 수정하면 됩니다 (코드 수정 불필요).

## 8. 시뮬레이터 (simulator.js)

실제 EC-FAN 하드웨어 없이 앱을 테스트할 수 있도록
가상 Modbus 슬레이브 장치를 소프트웨어로 구현했습니다.

```javascript
// 시뮬레이터 동작 방식
// 실제 시리얼 포트 대신 시뮬레이터에서 응답 생성
ModbusSimulator:
  - 65,536 레지스터의 가상 메모리 유지
  - Modbus RTU 프레임 수신 → 파싱 → 응답 생성 (50ms 지연)
  - FC03/FC04/FC06 등 주요 기능 코드 지원
  - 펌웨어 업데이트 시뮬레이션 (0x66 function code)
```

Settings 탭에서 "시뮬레이터 모드" 활성화 시 사용됩니다.

## 9. 자동화 테스트 프레임워크 (OSTestManager)

제조 검증 및 QC를 위한 자동화 테스트 시스템입니다.

```
os-test-manager.js    ← 테스트 실행 엔진
os-test-modbus.js     ← Modbus 통신 테스트 케이스
os-test-rs485.js      ← RS-485 물리 레이어 테스트
```

테스트 스텝 형식:
```javascript
{
  type: 'read_holding',     // 테스트 종류
  address: 0xD100,          // 대상 레지스터
  expected: slaveId,        // 예상 값
  description: 'Fan Address 확인'
}
```

Device Setup → Manufacture 탭에서 테스트를 실행하고 결과를 확인할 수 있습니다.
