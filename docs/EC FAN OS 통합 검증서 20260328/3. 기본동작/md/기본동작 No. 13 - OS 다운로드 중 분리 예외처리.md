# 기본동작 No. 13 - OS 다운로드 중 분리 예외처리

## 시험 항목

OS 다운로드 중 분리 예외처리

## 시험 목표

- OS 다운로드 진행 중 RS485 케이블을 강제 분리하여 통신이 끊어진 상황을 재현한다.
- 케이블 재접속 후 OS 재다운로드가 정상적으로 완료되고 펌웨어가 정상 적용됨을 확인한다.

## 시험 기준

### 사전 조건

- RS485 인터페이스를 통해 드라이브와 호스트 PC 연결 완료
- EC FAN OS 업데이트 툴 설치 완료
- 다운로드할 F/W 파일 준비 완료 (Main 또는 Inverter)

> ⚠️ 기본 통신 사양 (공장 출하 상태): Baudrate 19,200bps, Even Parity

### 시험 절차

1. RS485 인터페이스를 통해 드라이브와 호스트 PC를 연결한다.
2. 통신 툴을 실행하고 드라이브와 정상 통신되는지 확인한다.
3. EC FAN OS 업데이트 툴을 실행한다.
4. 통신 사양을 설정한 후 Connect 버튼을 누른다.
   - Baudrate: 19,200bps
   - Parity: Even
5. 다운로드할 F/W 파일을 선택한 후 Forced Download 체크박스를 체크한다.
6. Start 버튼을 눌러 다운로드를 진행한다.
7. **다운로드 중 RS485 케이블을 분리한다.**
   - 다운로드 진행 중간(예: 50% 진행 시점)에 케이블 강제 분리
8. 분리된 RS485 케이블을 다시 접속한 후 OS 업데이트를 다시 진행한다.
   - 툴에서 재연결 후 다시 Start
9. 전원 재투입 후 통신을 재접속하여 정상적으로 업데이트 되었는지 확인한다.
   - Main OS: SW Version [0xD003] 읽기 확인
   - Inverter OS: SW Version [0xD005] 읽기 확인

### 판정 기준

- OS 재다운로드하여 정상적으로 업데이트 되었을 경우 합격
- (Main OS의 경우 LED 점멸 주기 변화를 통해서도 업데이트 확인)

**합격 조건**: 위 판정 조건 모두 만족

시험 항목 : [RS485 Modbus 기반 OS 다운로드(Main/Inverter) 중 물리적 단선(Disconnection) 예외 주입 및 통신 스택/Bootloader 복합 자가 복구 통합 검증] 시험 목적 : EC FAN OS 업데이트 툴을 이용한 펌웨어(Main 및 Inverter) 다운로드 진행 중 RS485 통신 케이블 강제 분리 시, 드라이브 통신 스택이 무한 대기(Deadlock)에 빠지지 않고 Timeout 처리 후 수신 버퍼를 안전하게 비우는지 확인한다. 또한 케이블 재연결 직후(전원 유지)의 Handshake 복구 능력과, 단선 상태에서 전원까지 차단되는 복합 스트레스 환경(App 파손)에서의 Bootloader 생존 및 최종 OS 재다운로드 무결성을 종합적으로 검증한다.

시험 방법 : <Phase 1. 초기 통신 연결 및 버전 확인 (Base Setup)>

RS485 인터페이스를 통해 드라이브(Node ID: 0x01)와 PC를 연결하고, 통신(19200bps, Even)을 개방한다.
Main SW [0xD003] (01 04 D0 03 00 01 B8 4F) 및 Inverter SW [0xD005] (01 04 D0 05 00 01 F9 8E) 파라미터를 읽어 현재 버전을 기록한다.
EC FAN OS 업데이트 툴을 실행하고 드라이브와 Connect 한다.
<Phase 2. Main OS 다운로드 중 단선 예외 주입 및 Hot-Plug 통신 스택 복구 검증 (Main Disconnect Test)>

Main F/W 바이너리를 선택하고 'Forced Download' 체크 후 Start 버튼을 눌러 다운로드를 시작한다.
[예외 주입] 플래시 메모리에 데이터를 한창 기록 중인 시점(Progress Bar 약 30~50% 구간)에서, PC와 드라이브를 연결하는 RS485 통신 케이블을 기습적으로 완전히 분리(Unplug) 한다.
PC 업데이트 툴에서 통신 응답 없음(Timeout/Disconnect) 에러 팝업이 정상적으로 발생하는지 확인한다.
[Hot-Plug 스트레스] 드라이브 전원을 끄지 않은 상태 그대로 10초 대기 후, RS485 케이블을 다시 꽂는다(Re-plug).
툴에서 다시 Connect 및 Start를 눌렀을 때, 드라이브(Bootloader)가 꼬인 수신 버퍼를 비우고 새로운 다운로드 Handshake 요청을 정상적으로 수락하여 Main OS 재다운로드를 100% 완료하는지 확인한다.
<Phase 3. Inverter OS 다운로드 중 단선 예외 주입 및 듀얼 코어 IPC 복구 검증 (Inverter Disconnect Test)>

이번에는 Inverter F/W 바이너리를 선택하고 다운로드를 시작한다.
[예외 주입] Inverter OS 다운로드 도중(Progress 약 50%) 동일하게 RS485 케이블을 분리한다.
[IPC 병목 관찰] PC와 Main MCU 간의 통신이 끊겼을 때, Main MCU가 Inverter MCU로 데이터를 패스스루(Pass-through)하던 내부 통신(SPI/UART) 루프 역시 안전하게 타임아웃(Abort) 처리되는지 내부 상태를 가늠한다.
다시 케이블을 연결하고 재다운로드를 시도했을 때, Main MCU가 통신 먹통(Hang-up) 상태 없이 즉시 응답하며 Inverter OS 재다운로드를 끝까지 완수하는지 확인한다.
<Phase 4. 단선 + 전원 차단(Blackout) 복합 스트레스 주입 및 최악 조건 자가 복구 검증 (Worst-case Brick Prevention Test)>

다시 Main F/W (또는 Inverter F/W) 다운로드를 시작한다.
[복합 극한 스트레스] 다운로드 50% 진행 시점에서 RS485 케이블을 분리한다. 그 직후, 펌웨어가 깨진(Half-flashed) 상태의 드라이브 **주 전원을 강제로 차단(OFF)**해 버린다.
10초 대기 후 전원을 재투입하고, RS485 케이블을 다시 연결한다. (사용자의 당황한 대처 시나리오 재현)
통신 모니터링 툴(Modbus Master)로 Main SW [0xD003] (또는 Inverter SW [0xD005]) 읽기를 시도하여, App 영역 파손을 의미하는 0x0000 또는 0xFFFF가 응답되는지 확인한다.
OS 업데이트 툴을 연결하여 최종 재다운로드를 수행하고, 100% 완료 후 전원 재부팅 시 드라이브가 완벽하게 정상 구동 상태(LED 정상 점멸, 모터 제어 Ready)로 자가 복구되는지 최종 확인한다.
판정기준 : [Phase 1 판정 - 초기 연결]

툴과 장비 간의 Handshake가 Timeout 없이 한 번에 성공해야 하며 버전 정보가 정상 파싱되어야 한다.
[Phase 2 판정 - Main 단선 및 Hot-Plug 복구 (통신 스택 견고성 핵심)]

케이블 단선 시, 펌웨어의 수신 인터럽트(RX Interrupt)나 DMA가 쓰레기 데이터에 의해 데드락(Deadlock)에 빠지면 **불합격(Fail)**이다.
전원 리셋 없이 케이블 재연결만으로도 툴의 재시작(Start) 명령을 즉각 수용하여 다운로드를 처음부터 다시 시작(Pointer 초기화)할 수 있어야 합격 처리한다.
[Phase 3 판정 - Inverter 단선 및 듀얼 코어 IPC 복구]

단선 시 Main MCU가 내부적으로 Inverter MCU의 응답을 무한정 기다리는 로직 결함(Infinite Wait)이 없어야 한다.
케이블 재연결 후 Inverter OS를 다시 밀어 넣었을 때, Main MCU가 정상적으로 중계 역할을 재개하여 다운로드가 완료되어야 합격이다.
[Phase 4 판정 - 복합 에러 최악 조건 자가 복구 (가장 중요한 신뢰성 보증)]

통신이 끊기고 전원마저 날아간 최악의 App 훼손 상태에서도, Bootloader는 어떠한 경우라도 살아남아 RS485 통신을 개방해야만 한다.
복구 모드에서 읽어 들인 버전은 명확히 0x0000(또는 에러코드)을 리포트해야 하며, 툴을 통한 재다운로드 및 부팅 후 시스템이 100% 정상화(Brick 회피 성공) 되어야만 최종 합격 처리한다.
