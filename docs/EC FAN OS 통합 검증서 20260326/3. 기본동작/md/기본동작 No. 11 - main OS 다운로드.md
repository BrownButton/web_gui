# 기본동작 No. 11 - main OS 다운로드

## 시험 항목

main OS 다운로드

## 시험 목표

- EC FAN OS 업데이트 툴을 이용하여 Main F/W 파일을 드라이브에 다운로드한 후 전원 Off/On 시 새로운 OS가 정상 적용됨을 확인한다.
- 다운로드 후 Main SW Version [0xD003] 파라미터에 업데이트된 버전이 반영됨을 확인한다.

## 시험 기준

### 사전 조건

- RS485 인터페이스를 통해 드라이브와 호스트 PC 연결 완료
- EC FAN OS 업데이트 툴 설치 완료
- 다운로드할 Main F/W 파일 준비 완료
- 현재 탑재 버전과 다운로드할 버전이 상이함을 확인

> ⚠️ 기본 통신 사양 (공장 출하 상태): Baudrate 19,200bps, Even Parity
> 드라이브 설정이 다를 경우 접속 불가 — 사전에 현재 설정 확인 필요

### 시험 절차

1. RS485 인터페이스를 통해 드라이브와 호스트 PC를 연결한다.
2. 시험 전 현재 Main SW Version을 확인한다.

**Main SW Version 읽기 명령 — FC(04)**

| 바이트 | 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 |
|--------|------|------|------|------|------|------|------|------|
| 내용 | 0x01 | 0x04 | 0xD0 | 0x03 | 0x00 | 0x01 | CRC Lo | CRC Hi |

→ `01 04 D0 03 00 01 B8 4F`

3. EC FAN OS 업데이트 툴을 실행한다.
4. 통신 사양을 설정한 후 Connect 버튼을 누른다.
   - Baudrate: 19,200bps
   - Parity: Even
5. 다운로드할 **Main** F/W 파일을 선택한 후 Forced Download 체크박스를 체크한다.
6. Start 버튼을 눌러 다운로드를 진행한다.
   - 다운로드 진행 중 Progress Bar 모니터링
7. 다운로드 완료 후 전원을 Off/On한다.
8. 통신을 재접속하여 Main SW Version [0xD003]을 읽어 업데이트된 버전으로 변경되었는지 확인한다.
   - LED 점멸 주기 변화를 통해서도 업데이트 여부 확인 가능

### 판정 기준

- Main OS 다운로드 및 전원 Off/On 후 Main SW Version(0xD003)이 업데이트된 버전으로 확인됨
- (Main OS의 경우 LED 점멸 주기 변화를 통해서도 업데이트 확인 가능)

**합격 조건**: 위 판정 조건 모두 만족

시험 항목 : [RS485 Modbus 기반 Main OS 다운로드(OTA) 파일 무결성(CRC) 검증 및 플래싱 중단(Blackout) 시 Bootloader 안전 복구 통합 검증]

시험 목적 : EC FAN OS 업데이트 툴을 이용한 Main F/W 정상 다운로드 및 적용(버전 갱신/LED 점멸 변화)을 확인한다. 또한, 타겟이 다른 파일(Inverter F/W)이나 손상된 바이너리 주입 시의 방어 로직(Header/CRC Check), 그리고 플래싱(Flashing) 도중 전원/통신 단절 시 하드웨어가 벽돌(Brick)이 되지 않고 Bootloader 모드로 안전하게 진입하여 재다운로드를 대기하는지 종합적으로 검증한다.

시험 방법 : <Phase 1. 초기 상태 연결 및 현재 펌웨어/LED 상태 확인 (Base Setup)>

RS485 인터페이스를 통해 드라이브(Node ID: 0x01)와 PC를 연결하고 19200bps(Even) 통신을 개방한다.
Main SW Version [0xD003] 파라미터를 Read Input Register (FC: 0x04) 명령(01 04 D0 03 00 01 B8 4F)으로 읽어 현재 버전을 기록한다.
현재 Main OS가 정상 구동 중임을 알리는 상태 표시 LED의 점멸 주기(예: 1초 간격 토글, 1Hz)를 육안으로 확인한다.
<Phase 2. 정상 OS 다운로드 및 펌웨어 갱신 검증 (Positive OTA Test)>

EC FAN OS 업데이트 툴을 실행하고 Connect(19200bps, Even)를 누른다.
정상적으로 빌드된 상위 버전의 Main F/W 바이너리(.bin 또는 .hex) 파일을 선택하고 'Forced Download' 옵션을 체크한다.
Start 버튼을 눌러 다운로드를 진행하며, Progress Bar가 멈춤이나 통신 Timeout 에러 없이 100% 도달하는지 모니터링한다.
다운로드 완료 후 드라이브 전원을 OFF/ON(또는 자동 Soft-Reset) 한다.
재부팅 후 [0xD003]을 읽어 버전이 신규 버전으로 정확히 갱신되었는지 확인하고, 신규 OS에 반영된 상태 표시 LED의 새로운 점멸 주기(예: 0.5초 간격, 2Hz 빠른 점멸 등)가 정상 적용되었는지 확인한다.
<Phase 3. 잘못된 파일(손상/타겟 불일치) 주입 예외 검증 (Negative Binary Test)>

다시 업데이트 툴을 실행하되, 이번에는 Main F/W가 아닌 Inverter F/W 파일 또는 임의로 텍스트를 수정하여 CRC가 깨진 손상된 파일을 선택한다.
Start 버튼을 눌러 강제 다운로드를 시도한다.
Bootloader 파서(Parser)가 전송된 파일의 Header(Target ID) 또는 전체 CRC 무결성을 검사하여, 플래시 메모리 Erase(지우기)를 수행하기 전에 즉각 다운로드를 거부(Reject)하고 에러 팝업을 띄우는지 확인한다.
다운로드 실패 후 장비를 재부팅하여 기존의 Main OS가 훼손되지 않고 정상 동작하는지 교차 확인한다.
<Phase 4. 플래싱 중 전원/통신 단절 스트레스 및 Bootloader 자가 복구 검증 (Brick Prevention & Recovery Test)>

다시 정상적인 Main F/W 파일을 선택하고 다운로드를 시작한다.
[극한 스트레스 주입] Progress Bar가 약 50%를 지나며 플래시 메모리를 한창 굽고 있는 도중, 기습적으로 드라이브의 주 전원을 차단(Blackout)하거나 RS485 통신 케이블을 확 뽑아버린다.
10초 대기 후 전원/통신을 재연결한다. Main App 영역이 반쯤 지워져서(Corrupted) 정상 부팅이 불가능한 상태가 된다.
드라이브가 데드락(벽돌)에 빠지지 않고, 내부 Bootloader가 App 손상을 스스로 감지하여 **통신 대기 모드(Bootloader Mode, LED 상시 ON 또는 특정 에러 점멸 패턴)**로 안전하게 진입하는지 확인한다.
이 상태에서 업데이트 툴을 다시 연결하고, JTAG 등 별도의 복구 장비 없이 RS485 통신선만으로 Main F/W 다운로드를 재시도하여 100% 정상 복구(Recovery)가 완료되는지 최종 확인한다.

판정기준 :
[Phase 1 판정 - 초기 연결]
[0xD003] 버전을 읽었을 때 Exception 없이 정상 파싱되어야 하며, 시스템 LED가 스펙에 명시된 정상 패턴으로 동작해야 한다.

[Phase 2 판정 - 정상 업데이트 (기본 기능 핵심)]

다운로드 진행 중 Checksum 에러나 Timeout 응답 지연으로 인한 재시도(Retry)가 통신 스펙 허용치 내여야 한다.
재부팅 후 버전 파라미터가 신규로 변경되고, LED 동작이 신규 OS 릴리즈 노트의 사양과 정확히 일치해야 합격이다.

[Phase 3 판정 - 잘못된 파일 거부 (방어 로직 핵심)]

타겟이 다른 파일이나 깨진 파일을 넣었을 때 펌웨어(Bootloader)는 이를 식별해야 하며, 절대 기존 정상 App 영역을 지워버리는(Erase) 바보 같은 동작을 수행하면 안 된다.
즉시 NACK 또는 Exception을 반환하여 툴에서 'Invalid Firmware/CRC Error' 등의 명확한 실패를 리포트하고 기존 OS를 보호해야 합격 처리한다.

[Phase 4 판정 - 다운로드 중단 시 자가 복구 (가장 중요한 신뢰성 보증)]

OTA 중 전원/통신이 끊겼을 때 하드웨어가 먹통(Brick)이 되어 제조사로 입고해야 하는 상황이 발생하면 즉시 **불합격(Fail)**이다.
어떠한 경우라도 Bootloader는 독립적으로 살아남아 통신 포트를 열어두어야 하며, 재다운로드를 통해 기존 App 영역을 덮어쓰고 정상 상태로 100% 복구되어야만 최종 합격 처리한다.
