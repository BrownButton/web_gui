# OS 검증 리스트 작성 가이드 및 보완사항

> 현재 검증 리스트 분석 후 부족한 부분과 작성 시 유의사항을 정리한 문서

---

## 1. 현재 검증 리스트의 근본적인 문제

### 1-1. "성공 경로"만 검증하고 있음

현재 대부분의 항목이 아래 패턴을 따른다:

```
장비 연결 → 명령 전송 → 정상 응답 확인 → 합격
```

이것은 **기능이 존재하는지(Positive Test)** 만 확인하는 것이고,
**기능이 잘못된 상황을 제대로 처리하는지(Negative Test)** 는 거의 없음.

소프트웨어적으로 더 중요한 것은 **예외 상황을 얼마나 안전하게 처리하는가** 이다.

### 1-2. 단일 케이스만 검증

예: Modbus FC03 시험에서 `Set Point [0xD001]` 하나만 읽는다.
→ 다른 레지스터는 정상인지? 범위 밖 주소를 읽으면 어떻게 되는지? 확인 없음.

## 2. 분야별 보완이 필요한 항목

---

### 2-1. RS485 통신

#### 현재 부족한 부분

| 부족한 내용 | 보완 방향 |
|------------|----------|
| 정상 연결만 확인 | 잘못된 Baudrate로 접속 시도 → 연결 안 됨 확인 |
| Parity 변경 후 기본값 복구 확인만 | 잘못된 Parity로 접속 → 에러 응답 또는 무응답 확인 |
| Node ID 1개만 검증 | 경계값(ID=0, ID=247, ID=248 이상) 설정 시 동작 확인 |
| Broadcast 단순 전송 | Broadcast 후 특정 노드에 응답 요청 → Broadcast는 응답 없음 검증 |
| 통신 중단 상황 없음 | 통신 중 케이블 분리 후 재연결 → 통신 재개 확인 |

#### 추가해야 할 테스트 케이스

```
[실패 케이스]
- 잘못된 Baudrate(9600, 38400 등)로 접속 시도 → 응답 없음 확인
- 잘못된 Parity(None, Odd)로 접속 시도 → 응답 없음 확인
- Node ID를 0x00 (Broadcast ID)으로 설정하면 개별 응답 없음 확인
- Node ID를 248 이상으로 설정 시 드라이브 처리 방식 확인

[경계값 케이스]
- Node ID 최솟값(1)과 최댓값(247) 설정 및 통신 검증
- 동일 ID 2대 연결 시 충돌 처리 확인

[복구 케이스]
- 통신 케이블 분리 → 재연결 후 정상 통신 재개 확인
- 전원 Off 후 재투입 시 통신 파라미터(Baudrate, Parity) 유지 확인
```

### 2-2. Modbus RTU

#### 현재 부족한 부분

| 부족한 내용 | 보완 방향 |
|------------|----------|
| 정상 읽기/쓰기만 확인 | 잘못된 주소, 범위 초과값 등 에러 응답 코드 검증 |
| 에러 코드 존재만 언급 | Exception Code 0x01~0x06 각각의 발생 조건 및 응답 확인 |
| CRC 오류 처리 없음 | CRC가 틀린 명령 전송 시 드라이브 무응답 또는 에러 응답 확인 |
| FC03/04 각 1개 레지스터만 | 연속 레지스터 읽기(Quantity > 1) 동작 확인 |
| 읽기 전용 레지스터 쓰기 시도 없음 | Input Register(FC04 대상)에 FC06으로 쓰기 → Exception 응답 확인 |

#### 추가해야 할 테스트 케이스

```
[FC03 - Read Holding Register]
- 존재하지 않는 주소 읽기 → Exception Code 0x02 (Illegal Data Address) 응답 확인
- Quantity=0 또는 Quantity>125 전송 → Exception 응답 확인
- 연속 레지스터 다수 읽기 (Quantity=5~10) 정상 동작 확인

[FC04 - Read Input Register]
- 쓰기 전용 또는 존재하지 않는 Input Register 읽기 → Exception 응답 확인
- Motor Status [0xD011] 정지 상태 / 운전 중 / 에러 상태 각각의 값 비교

[FC06 - Write Single Register]
- 읽기 전용 레지스터에 쓰기 시도 → Exception Code 0x03 (Illegal Data Value) 또는 0x04 응답 확인
- 허용 범위를 벗어난 값 쓰기 → 드라이브의 값 클리핑 또는 에러 처리 확인
- 운전 중 파라미터 변경 → 즉시 반영 또는 에러 처리 확인

[FC10 - Write Multiple Registers]
- Byte Count와 실제 데이터 불일치 → Exception 응답 확인
- 일부는 유효, 일부는 무효 주소 포함 → 전체 롤백 또는 부분 적용 확인

[공통]
- CRC 오류 프레임 전송 → 무응답 또는 에러 응답 확인
- 같은 명령 연속 빠르게 전송(Stress) → 버퍼 오버플로우 없음 확인
- 응답 타임아웃(드라이브 응답 없을 때 마스터 처리) 확인
```

### 2-3. 기본동작

#### 현재 부족한 부분

| 부족한 내용 | 보완 방향 |
|------------|----------|
| Reset은 성공 경로만 | 운전 중 SW Reset 시 안전하게 정지 후 재시작하는지 확인 |
| Alarm Reset 조건 단순 | 알람 원인 미해소 상태에서 Reset → 즉시 재알람 발생 확인 |
| EEPROM Save 1회만 | 반복 Save 후 전원 재투입 → 최종 저장값 유지 확인 |
| 전류제한 동작 검증 없음 | Max Coil Current 설정 후 실제 전류가 제한되는지 파형/파라미터로 확인 |
| 온도 센서 정확도 미검증 | Board/IGBT 온도를 실측값(열화상 카메라 등)과 비교 |
| OS 다운로드 완료 판정 기준 불명확 | 버전만 비교? LED만 확인? → 판정 기준 구체화 필요 |

#### 추가해야 할 테스트 케이스

```
[Reset]
- 운전 중 SW Reset → 즉시 정지 후 재부팅 확인 (출력 차단 여부 확인)
- Reset 명령 여러 번 연속 전송 → 오작동 없음 확인
- Alarm 원인 미해소 상태에서 Alarm Reset → 재알람 발생 확인

[EEPROM Save]
- EEPROM Save 연속 반복(10회 이상) → 정상 동작 및 저장값 무결성 확인
- 저장 중 전원 차단(최악의 경우) → 부팅 후 이전 유효 값 복구 확인 (가능 시)
- Save 없이 파라미터 변경 → 전원 재투입 후 변경 전 값으로 복구 확인

[OS 다운로드]
- 잘못된 F/W 파일(다른 제품용) 다운로드 시도 → 거부 또는 에러 처리 확인
- 다운로드 완료 직후 전원 차단(타이밍 최악) → 재부팅 시 정상 또는 안전 상태 확인
- Main OS와 Inverter OS 동시 업데이트 시나리오 → 순서 및 안전성 확인
```

### 2-4. 구동동작

#### 현재 부족한 부분

| 부족한 내용 | 보완 방향 |
|------------|----------|
| 단일 속도/토크 지점만 검증 | 최솟값(0), 최댓값(65535/64000), 경계값 검증 |
| 정방향 구동만 확인 | 운전 중 방향 전환 시 안전한 처리 확인 |
| 모드 전환 정지 후만 확인 | 운전 중 Operating Mode 변경 → 즉시 반영 또는 에러 처리 확인 |
| Set Value Source 전환 없음 | RS485 → PWM 전환 시 지령 우선순위 처리 확인 |
| 가속/감속 특성 미검증 | 속도 급변 명령 시 ramping 또는 즉시 반영 여부 확인 |
| Closed-loop 추종 오차 미검증 | 명령 100rpm 시 실제 속도 허용 오차 범위 확인 필요 |

#### 추가해야 할 테스트 케이스

```
[Set Point 경계값]
- Set Point = 0 → 완전 정지 확인
- Set Point = 65535 (Open-loop 100%) → 최대 출력 확인
- Set Point = 64000 (Velocity 최대) → 최대 속도 확인
- Set Point를 음수(-1, 0xFFFF 초과) 전송 → 클리핑 또는 에러 처리 확인

[운전 중 파라미터 변경]
- 운전 중 Running Direction 변경 → 안전한 방향 전환(급정지 후 재구동 또는 서서히) 확인
- 운전 중 Operating Mode 변경 → 즉시 반영 시 출력 불연속 여부 확인
- 운전 중 Max Coil Current 변경 → 전류 제한 즉시 반영 확인

[Closed-loop 속도 제어]
- 목표 속도 100rpm → 실제 속도 허용 오차 범위(예: ±5rpm) 내에 안착하는지 확인
- 목표 속도 단계적 변경(100→200→500→100rpm) → 각 구간 추종 안정성 확인
- 부하 변동 시(팬 블레이드 있음/없음) 속도 유지 성능 확인

[Set Value Source 우선순위]
- RS485로 Set Value 설정 중 PWM 신호 입력 → 어느 쪽이 우선인지 확인
- Set Value Source 전환 시 지령 연속성 확인
```

### 2-5. 보호동작

#### 현재 부족한 부분

| 부족한 내용 | 보완 방향 |
|------------|----------|
| 에러 발생 확인만 | 에러 발생 → 출력 차단 → Alarm Reset → 정상 복귀까지 전 과정 검증 |
| 에러 원인 미해소 상태에서 Reset 없음 | 원인 유지 → Reset → 즉시 재알람 발생 확인 |
| 복수 에러 동시 발생 없음 | 2개 이상 에러 동시 발생 시 Motor Status 값 및 우선순위 확인 |
| 에러 임계값 경계 미검증 | 임계값 직전(Warning 영역)과 임계값 초과(Error 영역) 구분 확인 |
| 전원 재투입 후 에러 상태 지속 여부 | 에러 상태에서 전원 Off/On → 에러 클리어 여부 확인 |
| 보호 동작 후 출력 차단 여부 | 에러 발생 즉시 모터 출력 차단되는지 확인 |

#### 추가해야 할 테스트 케이스

```
[에러 처리 전 과정]
에러 발생 → Motor Status 확인 (에러 비트 Set)
         → 모터 출력 차단 확인 (Actual Speed = 0 또는 무응답)
         → 에러 원인 해소
         → Alarm Reset 명령 전송
         → Motor Status 확인 (에러 비트 Clear)
         → 정상 구동 명령 전송 → 정상 동작 확인

[에러 원인 미해소 Reset]
1. 에러 조건 유지 상태에서 Alarm Reset 전송
2. Motor Status 확인 → 즉시 재알람 발생 여부 확인
3. (에러가 클리어되면 안 됨 → 원인 해소 전 Reset이 의미 없음을 검증)

[복수 에러]
- 주 전원 결상(PHA) + 모터 구속(BLK) 동시 발생 → Motor Status 값 확인
  (예: 0x0091 = PHA + FB + BLK)
- 에러 코드가 어느 것으로 표시되는지 우선순위 확인

[전원 재투입과 에러 상태]
- 에러 발생 상태에서 전원 Off → On → 에러 자동 클리어 여부 확인
- 에러 원인이 지속되면(예: 자석 제거 상태) 전원 재투입 후에도 에러 발생 확인

[임계값 경계]
- 저전압 에러 임계값(예: 220Vac)에서 ±10V 구간 테스트
  - 230Vac: 에러 없음 확인
  - 220Vac: 에러 발생 확인
  - 215Vac: 에러 지속 확인
- 과속 에러: Fan Blade 제거 상태에서 Set Point를 서서히 올려 에러 발생 속도 측정
```

## 3. 전반적으로 빠진 테스트 유형

### 3-1. Negative Test (실패해야 하는 케이스)

현재 거의 모든 테스트가 **성공 경로만** 다루고 있음.

| 테스트 유형 | 예시 |
|------------|------|
| 잘못된 입력 | 범위 초과 파라미터 값 설정 시 드라이브 처리 방법 |
| 잘못된 프로토콜 | CRC 오류, 잘못된 Function Code, 부족한 바이트 수 |
| 잘못된 순서 | 정지 명령 없이 모드 전환, 에러 미해소 상태에서 구동 명령 |
| 잘못된 상태 | 에러 상태에서 구동 명령 전송 → 거부 또는 에러 유지 확인 |

### 3-2. Boundary Value Test (경계값)

| 파라미터 | 최솟값 | 최댓값 | 경계 초과 |
|---------|--------|--------|----------|
| Set Point | 0 | 65535 | 65536 이상 전송 시 처리 |
| Node ID | 1 | 247 | 0, 248 이상 설정 시 처리 |
| Baudrate | 9600 | 115200 | 지원하지 않는 값 설정 시 처리 |
| Max Coil Current | (최솟값) | (최댓값) | 범위 초과 설정 시 처리 |

### 3-3. State Transition Test (상태 전환)

드라이브는 다양한 상태가 있는데 상태 전환 시나리오가 거의 없음.

```
[주요 상태]
정지 → 운전 → 정지 (기본)
정지 → 에러 → Reset → 정지 (기본)
운전 → 에러 → Reset → 운전 (검증 필요)
운전 → 방향전환 → 운전 (검증 필요)
운전 → 모드전환 → 운전 (검증 필요)
에러 → 에러(원인 미해소) → Reset 시도 → 에러 유지 (검증 필요)
```

### 3-4. Stress / Repetition Test (반복 내구성)

| 테스트 | 내용 |
|--------|------|
| 반복 구동/정지 | 구동 → 정지를 N회 반복 후 파라미터 이상 없음 확인 |
| 반복 EEPROM Save | Save를 N회 반복 후 저장 무결성 확인 |
| 장시간 폴링 | Modbus 폴링을 수 시간 지속 후 통신 안정성 확인 |
| 반복 에러/Reset | 에러 발생 → Reset을 N회 반복 후 정상 동작 확인 |

### 3-5. Concurrent / Timing Test (동시성/타이밍)

| 상황 | 검증 포인트 |
|------|-----------|
| 폴링 중 쓰기 명령 | 폴링 응답 중 쓰기 명령 전송 → 버스 충돌 없음 확인 |
| 전원 투입 중 통신 | 전원 투입 직후 즉시 Modbus 명령 → 응답 안정성 확인 |
| 운전 중 전원 차단 | 운전 중 전원 Off → 재투입 후 정상 상태 확인 |

## 4. 검증 리스트 작성 시 유의사항

### 4-1. 판정기준을 구체적으로 작성할 것

**나쁜 예시:**
```
- 정상으로 나타나면 합격
```

**좋은 예시:**
```
- Motor Status [0xD011] 값이 0x0050 (4번, 6번 비트 Set) 일 경우 합격
- Actual Speed [0xD02D] 값이 목표 속도 ±5% 이내일 경우 합격
- EEPROM Save 후 전원 재투입 시 설정값이 ±0 오차로 일치할 경우 합격
```

### 4-2. 시험 결과 기록란을 구체화할 것

**나쁜 예시:**
```
- [ ] 합격 / 불합격
```

**좋은 예시:**
```
| 케이스 | 기대값 | 실측값 | 판정 |
|--------|--------|--------|------|
| Set Point = 3200, Actual Speed | 100 RPM | ___ RPM | 합격/불합격 |
| Set Point = 6400, Actual Speed | 200 RPM | ___ RPM | 합격/불합격 |
```

### 4-3. 초기 상태(Precondition)를 명시할 것

각 시험 시작 전 드라이브가 어떤 상태여야 하는지 명시:
```
[시험 전 조건]
- Operating Mode: Open-loop (0x0002)
- Set Point: 0
- 알람 없음 (Motor Status = 0x0000)
- Node ID: 1
```

### 4-4. 복구 절차를 반드시 포함할 것

특히 물리적 변형이 필요한 시험(자석 제거, 와셔 삽입, 케이블 분리)은 반드시 복구 절차와 복구 후 정상 동작 확인을 포함해야 함.

### 4-5. 에러 상태의 전 과정을 기록할 것

에러 시험은 아래 4단계를 모두 포함해야 함:
```
1. 에러 조건 유발 → Motor Status 확인 (에러 비트 Set 확인)
2. 출력 차단 확인 (모터 정지 확인)
3. 에러 원인 해소 → Alarm Reset 명령 전송
4. 정상 복귀 확인 (Motor Status 정상, 구동 가능 확인)
```

## 5. 우선순위별 보완 항목

### 즉시 보완 필요 (High)

1. **Modbus Exception Code 검증** — 존재하지 않는 주소, 범위 초과값에 대한 에러 응답 확인
2. **에러 복구 전 과정 검증** — 현재 "에러 발생 확인"만 있고 "복구 과정"이 없음
3. **에러 원인 미해소 상태에서 Reset** — 에러가 유지되어야 함을 검증
4. **Set Point 경계값(0, 최댓값)** — 현재 10%, 50% 같은 중간값만 검증 중
5. **EEPROM Save 없이 전원 재투입** — 변경값이 초기화되는지 명시적 검증

### 추후 보완 (Medium)

6. **운전 중 파라미터 변경** — 방향, 모드, 전류제한 등 운전 중 변경 시 처리
7. **복수 에러 동시 발생** — Motor Status 비트 조합 확인
8. **임계값 경계 테스트** — 에러 발생 직전/직후 값 테스트
9. **통신 케이블 분리 후 재연결** — 통신 재개 안정성

### 환경 구축 후 진행 (Low)

10. **Stress 테스트** — 반복 구동/정지, 장시간 폴링
11. **동시성 테스트** — 폴링 중 쓰기 명령 충돌 방지
12. **저온 환경 테스트** — IPM Low Temperature (챔버 필요)

## 6. 검증서 샘플 — 보완된 형식

아래는 `5-20. Error 0x58 Motor Shaft is Blocked` 시험을 보완된 형식으로 재작성한 예시:

```markdown
## 시험 전 조건
- Operating Mode: Open-loop (0x0002)
- Set Point: 0 (정지 상태)
- Motor Status [0xD011] = 0x0000 (알람 없음)

## 시험 방법

### [케이스 1] 정상 구동 (베이스라인)
1. Set Point = 6554(10% 토크) 설정
2. Actual Speed 확인 → 0보다 큰 값 (팬 회전 중)
3. Set Point = 0으로 정지

### [케이스 2] 구속 상태에서 구동 (에러 유발)
4. 팬 블레이드를 테이프로 구속
5. Set Point = 6554(10% 토크) 설정
6. 약 2~3초 대기
7. Motor Status [0xD011] 읽기 → 0x0090 (Bit4 FB + Bit7 BLK) 확인
8. Actual Speed [0xD02D] 읽기 → 0 (출력 차단 확인)

### [케이스 3] 에러 원인 미해소 상태에서 Reset (부적절한 Reset)
9. 구속 유지 상태에서 Alarm Reset 명령 전송
10. Motor Status 읽기 → 에러 비트가 재발생하는지 확인
    (판정: 에러가 즉시 재발생해야 정상)

### [케이스 4] 정상 복구
11. 구속 해제 (테이프 제거)
12. Alarm Reset 명령 전송
13. Motor Status [0xD011] 읽기 → 0x0000 확인 (에러 클리어)
14. Set Point = 6554 설정 → 정상 구동 확인

## 판정기준
| 케이스 | 기대 결과 | 판정 |
|--------|----------|------|
| 케이스 1 | Actual Speed > 0 (팬 회전) | 합격/불합격 |
| 케이스 2 | Motor Status = 0x0090, Actual Speed = 0 | 합격/불합격 |
| 케이스 3 | 에러 즉시 재발생 (재구동 불가) | 합격/불합격 |
| 케이스 4 | Motor Status = 0x0000, 정상 구동 가능 | 합격/불합격 |
```

## 7. 요약

현재 OS 검증 리스트의 가장 큰 문제는:

1. **"된다"를 확인하지, "잘 안 되는 상황에서도 올바르게 처리되는지"를 확인하지 않음**
2. **단일 값만 테스트하고 경계값, 범위 초과, 잘못된 입력을 테스트하지 않음**
3. **에러 발생 → 복구 전 과정을 하나의 시나리오로 검증하지 않음**
4. **판정기준이 주관적이고 구체적인 수치가 없음**

소프트웨어 검증의 목적은 **소프트웨어가 올바르게 동작하는가** 뿐만 아니라
**잘못된 상황에서도 안전하게 처리하는가** 를 포함해야 한다.
